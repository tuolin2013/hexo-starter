### 引言：老兵的窘境与工具的妥协

当决定重启 IP **“程序员老庹”** 时，我的第一道难关不是代码，而是工具：我没有一台可用的 PC。传统的 Hexo 部署依赖本地 `hexo generate` 命令，这让我陷入了 **“工具依赖”** 的困境。

我的目标非常明确：**用一部 Android 手机，完成博客的搭建、部署和日常内容发布。** 这篇文章，就是对这场 **“极限生存战”** 的全过程复盘。

---

### 第一阶段：环境初始化与快速启动（用云端替代本地）

我放弃了本地环境，将所有操作推向了云端：**GitHub 作为代码仓库**，**Vercel 作为自动化构建和部署平台**。

1.  **选择基础模板：** 我没有从零开始，而是 **Fork** 了一个功能完整的 Hexo 模板仓库。这避免了我在手机上处理复杂的初始化配置。
2.  **Vercel 快速部署：** 登录 Vercel，通过 **“Import Project”** 绑定我的 GitHub 仓库 `Laoto_blog`。
3.  **配置构建命令：** Vercel 并不支持 Hexo，需要手动配置。在 Vercel 设置中，我将构建命令配置为：`npm install && ./node_modules/.bin/hexo generate`。
4.  **构建输出目录：** 将输出目录（Output Directory）设置为 Hexo 默认的 **`public`**。

### 第二阶段：解决致命的“依赖缺失”与“构建失败”

第一次部署失败是必然的。我遇到的核心问题是 Vercel 找不到依赖，导致构建失败。

#### 1. 致命的 `package.json` 缺失

Vercel 依赖根目录下的 `package.json` 文件来知道需要安装哪些 Node.js 依赖。我的 Fork 模板里可能缺失或不完整。

* **解决方案：** 在手机浏览器中，进入 GitHub 仓库根目录，手动 **“Create new file”**（创建新文件），粘贴入 Hexo 必需的 `package.json` 基础配置。

#### 2. “Output Directory is empty” 错误排查

即使依赖安装成功，Vercel 仍然报错 **“The Output Directory public is empty”**。这说明 Vercel 无法执行 `hexo generate` 命令。

* **解决方案：** 经过排查，发现是 Vercel 构建系统在执行 `./node_modules/.bin/hexo generate` 时，路径权限或执行环境存在问题。最终通过 **简化文件名、确保所有文件结构完整**（尤其是 `scaffolds` 和 `source` 目录）后，通过 Vercel 的多次重试和内部优化，最终成功触发了 `hexo generate`，生成了 **`public`** 目录，部署成功。

### 第三阶段：内容生产的极简工作流

一旦部署成功，我的手机写作流程变得极其简单和高效。

1.  **写作工具：** 选用手机上的 **Markor** 或其他纯文本 Markdown 编辑器，专注于写作，避免视觉干扰。
2.  **文章发布：** 在手机浏览器中，直接进入 GitHub 仓库的 **`source/_posts/`** 目录，**创建新文件**（例如 `03-mobile-deployment.md`），粘贴 YAML 头部和正文内容。
3.  **排版控制：** 为保证首页的简洁性，我在 YAML 头部之后立即插入 **``** 标记，实现 **“只显示标题，点击再展开”** 的归档效果。
4.  **持续集成：** 提交（Commit）后，Vercel **自动拉取代码，自动构建，自动发布**。整个流程耗时不到 2 分钟。

### 结语：老兵对“工具依赖”的清醒

这场“手机部署战”的胜利，不仅仅是技术上的，更是哲学上的。

它证明了我们程序员最需要的，不是最强大的 IDE 或最快的 PC，而是对**核心技术原理的掌握**和**对工具的清醒认知**。

**架构的本质，是解决问题。** 既然我能在电瓶车上找到最优路径，就能在手机上找到最短的代码部署路径。这是我的 **“老兵重启”** 哲学的一部分：**用最少的资源，实现最大的健壮性（Robustness）和效率。**
